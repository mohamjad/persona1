1. The Problem
Most people know what they want to say. They do not know how it will land.
Every message is a move in a game played partially blind. The sender cannot see the board from the other side. They do not know if their tone reads as confident or arrogant, warm or desperate, direct or cold. They hit send and wait.
Every tool built to solve this — Grammarly, Rizz, Keys.ai, every AI rewriter — only models the sender. They make your words cleaner. None of them model the recipient. None of them show the downstream consequences of what you are about to send.
The result: people either say nothing because they are unsure how it will land, say something wrong and lose the moment, or copy templates that do not sound like them and feel fake to both parties.
persona1 solves a different problem: not how to write better, but how to see the conversation before you are in it.
2. The Product
2.1 Core Loop
persona1 is a Chrome extension that reads the conversation you are in — the recipient's profile, message history, platform context — and renders a chess-style move tree before you respond.
You type a draft. persona1 shows three predicted branches: if you send this, the conversation is likely to go here. You see where each path leads. You pick your move with full board visibility.
The output is not a rewrite. It is a map. The user decides. persona1 provides the intelligence.
The single biggest product risk: fake game tree. Branches that are obviously generic or predicted responses that feel like fanfiction will destroy trust on first use. Every prompt engineering decision in this spec is oriented around one rule: specificity over confidence. A slightly wrong specific prediction is more useful than a vague accurate one. The system must appear to be modeling the other person, not hedging.

2.3 The Chess UI — Exactly
Three options max. Not five. Not a menu. Three.
The recommended option is visually dominant — full card, highlighted. The other two are secondary. Each card renders:
The message — in the user's voice, specific to this moment, not a template
Predicted response — what the recipient is likely to say back. Must be specific enough to be possibly wrong.
Branch path — where the conversation goes in two moves. One sentence.
Why it works — one line, the strategic logic. Collapsed by default.
Risk — one line, amber flag if goal alignment < 60. Collapsed by default.
Users can copy any card or click Use This to insert directly into the compose box. Keyboard shortcuts: 1, 2, 3 to select. Enter to copy. That is the entire interaction model.
2.4 The Mirror Feature
After 25 interactions, persona1 begins surfacing patterns about the user's communication that they have not noticed. Not as a report. As observations that emerge when the data supports them.
'You hedge before every direct ask. It costs you.'
'You bury the point in professional messages. Your first sentence is rarely your real ask.'
'You are funnier in casual contexts than professional ones. You suppress it at work.'
Mirror insights are the retention hook. The product becomes a communication coach that knows the user specifically. Switching cost grows every week. After three months the persona model is irreplaceable.

2.6 Virality Mechanic
The chess tree visualization is the screenshot. When a user shares a persona1 output they are sharing a conversation map — a visually novel, immediately legible artifact that makes the product self-explaining.
Secondary mechanic: communication scorecards. After a conversation, persona1 rates the user's actual performance across confidence, clarity, strategic alignment, and emotional read. Users share scores the same way they share personality test results. Identity content that markets the product passively.
3. Passive Learning Architecture
3.1 Philosophy
There is no onboarding form. There is no setup. The extension installs, watches the user's first real conversations, and builds the persona profile from observed behavior.
Self-reported data is biased. People describe who they want to be. Observed behavior is the truth. A user who says they are 'direct' but hedges in every draft gets a persona profile that reflects the hedge, not the self-image. That is what makes the outputs feel accurate.
Cold start: on first install, the user selects one primary context (dating, professional, general) with a single tap. That seeds the situation preset default and gives the inference model a prior. Everything else is observed from that point forward. Zero forms.
3.2 The Three Learning Phases
Phase 1 — Observation Mode (uses 1-5)
The model has only the cold-start prior. Branches are generated with lower personalization confidence. The system logs everything silently: full draft text, what the user edited before sending, which option they chose, platform, preset, recipient context hash. No output is shown to the user about this. They just use the product.
Phase 2 — Active Calibration (uses 6-25)
Enough signal exists to begin personalizing. The persona profile JSON is populated field by field from observed patterns. The branch generation prompt starts injecting persona-specific constraints. Outputs begin to sound noticeably different from generic rewrites. The model starts flagging when the user's draft is working against their own patterns.
Phase 3 — Mirror Activation (use 25+)
The mirror prompt runs. Pattern thresholds are checked. When a pattern has 5+ data points, it is surfaced as an insight. The persona profile is now version-controlled — every update increments the version field and diffs are logged. The model is the user's. Exporting or losing it is a real loss.
3.3 Observation Pipeline
The content script runs a MutationObserver on every compose context it detects. It logs:
draft_raw — the text before any edits
draft_final — the text after edits, before send
delta — computed diff between raw and final. This is the highest-signal field. It shows what the user instinctively removes or adds.
option_chosen — which branch they selected (1, 2, or 3)
option_rejected — which they dismissed first
platform — extracted from URL pattern
recipient_context_hash — one-way hash of recipient signals. Not PII. Just a stable identifier for 'this type of person'.
session_id — groups interactions in the same sitting
All observation data is stored locally first in Chrome extension storage. It is synced to backend only when the user is on a paid plan. Free tier users have local-only persona profiles.

4. AI Inference Pipeline
4.1 Pipeline Architecture
Every persona1 inference call is a four-stage pipeline. Each stage is a separate prompt call. Stages 1 and 2 run in parallel. Stage 3 depends on both. Stage 4 is async, post-interaction.
Stage
Description
1 — context_extractor
Reads page DOM. Extracts recipient signals. Outputs recipient context JSON. Runs on page load, cached per session.
2 — persona_loader
Reads persona profile from storage. Applies platform calibration. Outputs calibrated persona JSON for this context.
3 — branch_generator
Core call. Receives draft + recipient context + calibrated persona + situation preset. Outputs 3 branch objects.
4 — persona_updater
Async. Runs after outcome signal. Receives current persona + interaction data + outcome. Outputs updated persona JSON. Increments version.
5 — mirror_trigger
Async. Runs at interaction_count % 25 === 0. Receives full interaction log. Outputs 1-3 insight strings if pattern thresholds met.

4.3 Stage 3 — branch_generator Prompt
This is the core call. The fake game tree risk lives entirely in this prompt. The specificity rule is enforced here.
SYSTEM:
You are the inference engine for persona1, a conversation intelligence system.
You receive:
- A calibrated model of the sender: how they communicate, their patterns, their gaps
- A real-time model of the recipient: extracted from the live conversation
- The sender's current draft
- The situation preset (date / pitch / negotiate / etc)
Your job: generate exactly 3 message options, each with a predicted conversation branch.
CRITICAL RULES:
1. Every option must sound like the sender. Not generic AI. Not a template.
If the persona profile shows they write in fragments, write in fragments.
If they avoid exclamation marks, do not use them.
2. Predicted responses must be SPECIFIC. A prediction like 'she might respond
positively' is a failure. A prediction like 'she will probably deflect with
humor and change the subject' is a success. Be specific enough to be wrong.
3. Branch paths must name real downstream consequences, not vague directions.
4. If the user's draft is actively working against their stated goal,
flag it explicitly in draft_warning. Do not silently fix it.
5. Option 1 is the recommended option. It is not always the safest.
It is the one most aligned with the goal given the full context.
6. Never moralize. Never add disclaimers. Show the board.
Output: valid JSON array of exactly 3 objects.
{
"option_id": 1 | 2 | 3,
"is_recommended": boolean,
"message": string,
"predicted_response": string,
"branch_path": string,
"goal_alignment_score": 0-100,
"why_it_works": string,
"risk": string | null
}
Also output at root level:
{
"draft_warning": string | null,
"branches": [option1, option2, option3]
}
SENDER PERSONA: {PERSONA_JSON}
RECIPIENT CONTEXT: {CONTEXT_JSON}
SITUATION: {PRESET}
DRAFT: {USER_DRAFT}
4.4 Stage 4 — persona_updater Prompt
SYSTEM:
You are the persona update kernel for persona1.
You receive the current persona profile JSON, a completed interaction,
and an outcome signal (positive / neutral / negative).
Update the persona JSON based on what this interaction reveals.
Rules:
- Be conservative. One data point does not override an established pattern.
- Only update fields where this interaction adds meaningful signal.
- Increment version by 1.
- Update last_updated to current ISO timestamp.
- Increment interaction_count by 1.
- Update context_performance for the relevant preset.
- If a new observed_pattern is evident, add it with count: 1, confidence: 0.4.
- If an existing pattern was confirmed, increment count and recalculate confidence.
- If an existing pattern was contradicted, decrement confidence. Remove if < 0.3.
Output: updated persona JSON only. No commentary.
CURRENT_PERSONA: {PERSONA_JSON}
INTERACTION: { preset, draft_sent, option_chosen, recipient_context_hash }
OUTCOME: "positive" | "neutral" | "negative"
4.5 Stage 5 — mirror_trigger Prompt
SYSTEM:
You are the mirror kernel for persona1.
You receive a user's full interaction log: their drafts, chosen options,
rejected options, outcome signals, and the delta between what they wrote
and what they sent.
Your job: surface 1-3 communication patterns this person has not noticed.
Rules:
- Only surface patterns with 5+ supporting data points.
- Be specific. Not 'you hedge sometimes.' Say 'you hedge before every direct ask.'
- Frame as observation, not advice. Do not tell them what to do.
- One sentence per insight. No softening. No praise sandwich.
- If no pattern meets the 5-point threshold, return an empty array.
- Do not invent patterns. Only report what the data shows.
Output: JSON array of insight strings. May be empty.
INTERACTION_LOG: {LOG_JSON}

7.2 Cost Control
Monthly tier: $6 API cost cap per billing period. Above this threshold the system throttles response latency — not quality. A soft upgrade nudge appears. No hard cutoff. Hard cutoffs create churn. Throttling creates upgrade motivation.
At $9/month with a $6 cost cap, gross margin per user is $3 before infrastructure overhead. At volume, Anthropic API pricing improves via negotiation. Prompt caching on the persona profile (same system prompt across calls) reduces token cost significantly.
7.3 Paywall Gate
Free tier uses are counted in Chrome local storage at p1_usage_count. The background service worker checks this before every /v1/analyze call. On the 4th attempt, the Stripe checkout flow is triggered inline in the sidebar. No redirect. No new tab. Stripe embedded checkout inside the extension sidebar.

9. Risk Register

CRITICAL

HIGH
Conservative update rules in persona_updater. Confidence thresholds. Pattern requires 5+ data points before surfacing. User can view and edit persona profile at any time.

HIGH
