Section 2  ·  The Scoring Problem
Why the annotation is disconnected and the full fix

2.1 What Is Wrong
The current annotation layer is a standalone process that evaluates the user&apos;s draft message independently of everything else the system knows. It does not receive the persona profile. It does not receive the recipient context. It does not know the situation preset or the user&apos;s goal for this conversation. It is essentially a sentiment and grammar checker wearing a chess annotation costume.
This is why sending a perfectly calibrated message still gets a ?? — the system has no idea what &apos;correct&apos; looks like for this specific user talking to this specific person in this specific situation. The symbol is meaningless because it has no reference point.

The fix is architectural, not a prompt improvement. The annotation layer must become a downstream consumer of the full pipeline context — the same inputs that branch_generator receives — not a parallel process with none of that context.

2.2 The Target Architecture: Parameterized Scoring
The right architecture splits the scoring into two layers that run at different times with different costs.

Layer 1 — Parameterization (once per conversation, ~1 API call)
When persona1 detects a compose context and establishes the conversation parameters, one LLM call runs that outputs a scoring config for this specific conversation. This config is cached for the session. It costs tokens once, not on every keystroke.

// Parameterization output schema
{
  &quot;primary_goal&quot;: &quot;get a yes to the meeting&quot;,
  &quot;conversation_phase&quot;: &quot;close&quot;,
  &quot;tone_target&quot;: &quot;warm-direct&quot;,
  &quot;blunder_patterns&quot;: [
    &quot;apology opener&quot;,
    &quot;defensive language&quot;,
    &quot;too many options given&quot;,
    &quot;over-explanation of reasoning&quot;
  ],
  &quot;mistake_patterns&quot;: [
    &quot;hedging before direct ask&quot;,
    &quot;passive voice on the ask&quot;
  ],
  &quot;interesting_patterns&quot;: [
    &quot;reframe of their concern as your feature&quot;,
    &quot;callback to earlier in thread&quot;
  ],
  &quot;brilliant_patterns&quot;: [
    &quot;direct ask with single reason, no alternatives&quot;,
    &quot;setup that makes them ask the question you want&quot;
  ],
  &quot;recipient_risk_factors&quot;: [
    &quot;responds negatively to pressure&quot;,
    &quot;high formality preference&quot;
  ],
  &quot;persona_risk_factors&quot;: [
    &quot;user tends to hedge before asks&quot;,
    &quot;user over-explains under uncertainty&quot;
  ]
}

Layer 2 — Real-Time Deterministic Scoring (every keystroke, zero API calls)
The rule engine takes the session scoring config and evaluates the current draft against it locally. No network call. No tokens. Runs in under 5ms. Returns a symbol and a one-line reason.

Library
json-rules-engine (MIT) — TypeScript rule engine that takes a JSON fact set and evaluates against a JSON rule config. Exactly right for this pattern.
Input
{ draft, scoring_config, persona_patterns, draft_length_vs_baseline }
Output
{ symbol: &apos;!!&apos; | &apos;!&apos; | &apos;!?&apos; | &apos;?&apos; | &apos;??&apos; | &apos;-&apos;, reason: string, confidence: 0-1 }
Trigger
Debounced 100ms on every keystroke in the compose box
Fallback
If scoring_config not yet loaded, show &apos;-&apos; (neutral). Never block the user.

Layer 3 — LLM Arbitration (only when deterministic is ambiguous)
When the rule engine returns confidence < 0.6 or conflicting signals, a lightweight LLM call fires. It receives the deterministic score as input plus the draft and asks a narrow question — not &apos;is this a good message&apos; but &apos;given these specific signals, confirm or override the annotation.&apos; This keeps the LLM in the loop for edge cases without making it the primary evaluator.

2.3 The Parameterization Prompt
This prompt fires once when the compose context loads. It receives persona profile + recipient context + situation preset. It outputs the scoring config that drives the entire annotation session.

SYSTEM:
You are the scoring configuration engine for persona1.
You receive: the sender&apos;s calibrated persona profile, the extracted recipient context,
and the situation preset for this conversation.

Your job: output a scoring config that the real-time annotation engine will use
to evaluate every draft this user sends in this conversation.

This config must reflect:
- What success looks like for this specific goal with this specific person
- What patterns from the sender&apos;s profile are liabilities in this context
- What the recipient&apos;s signals suggest they react negatively to
- What moves are brilliant vs blunder specifically for this situation type

Be specific. Generic patterns (e.g. &apos;be confident&apos;) are not useful.
Name exact linguistic and structural patterns that the rule engine can check.

Output ONLY valid JSON matching the ScoringConfig schema. No commentary.

PERSONA: {PERSONA_JSON}
RECIPIENT_CONTEXT: {CONTEXT_JSON}
PRESET: {PRESET}

2.4 Training Data for Non-Robotic Scoring
The parameterization prompt should be augmented with a curated few-shot library of real conversation examples. Not fine-tuning — a retrieved example set that the prompt samples based on the detected conversation type. This is what makes the scoring feel like it understands conversation dynamics rather than just text patterns.

Few-shot library sources
Negotiation transcripts — Never Split the Difference case studies, real recorded sales calls, FBI hostage negotiation examples
Dating conversation analysis — documented conversations with outcome annotation, academic literature on attraction and rapport-building
Professional communication — email case studies, pitch recordings with known outcomes
Conflict resolution — mediation transcripts, difficult conversation frameworks (Crucial Conversations, Radical Candor)

These get embedded, stored in pgvector, and retrieved by conversation type at parameterization time. The prompt receives 3-5 relevant examples that ground the scoring config in real outcomes, not abstract principles.

Storage
pgvector on Cloud SQL — embed each example, retrieve top-k by cosine similarity to current preset + recipient type
Embedding model
text-embedding-3-small via OpenRouter — cheap, accurate for semantic similarity
Retrieval trigger
Parameterization call — retrieve before generating scoring config, inject as few-shot examples
Library size
Start with 50-100 curated examples per major preset. Grows over time from user outcomes.
GCP storage
pgvector table in existing Cloud SQL instance. No new infrastructure.

Section 3  ·  UX Architecture
Everything wrong with the current feel and the full stack to fix it

Source: Premium UI infrastructure research report. The three highest-leverage UX changes are: (1) Shadow DOM isolation, (2) deliberate motion language with Motion, (3) local caching for perceived zero latency. Everything else is polish on top of these three.

3.1 The Three Immediate Fixes
Fix 1 — Shadow DOM Isolation (highest leverage)
The single biggest cause of clunky is CSS collision. Host page styles break the sidebar. Extension styles break the host page. This is not fixable with better CSS — it requires architectural isolation.

Pattern
Shadow DOM for all injected UI — orb dock, branch tree, annotation glyph, settings popup
Blueprint
WXT createShadowRootUi (MIT) — explicit ShadowRoot UI mode with cssInjectionMode: &apos;ui&apos;
Alternative
Iframe injection for sites with extremely hostile CSS. Use as fallback, not default.
Key constraint
Portals and popovers must mount inside the ShadowRoot, not document.body. Radix primitives need configuration for this.
Result
Spacing, fonts, colors become consistent everywhere. Random brokenness disappears.

shadowRoot.attachShadow({ mode: &apos;open&apos; })
document.body.appendChild(shadowRoot)

Fix 2 — Motion Language (Motion library)
AutoAnimate is already in the repo. It handles incidental list transitions well. It cannot express the deliberate motion language the product needs. Keep AutoAnimate for quiet transitions. Add Motion for hero interactions.

Library
motiondivision/motion (MIT) — hybrid WAAPI + JS engine, springs, stagger, layout-aware transitions
Dock open/close
Spring animation. Not linear. Not ease-in-out. Spring with stiffness 300, damping 25. Feels physical.
Branch card reveal
Staggered entrance. Card 1 at 0ms, card 2 at 60ms, card 3 at 120ms. Each slides up 8px and fades in.
Annotation glyph appearance
Scale from 0.7 to 1.0 with spring. Duration 180ms. Weight feels intentional.
Selection confirmation
Brief highlight flash on selected card. 200ms. Then sidebar collapses.
Loading shimmer
CSS-only skeleton shimmer on branch cards while prefetch is running.
Bundle note
Motion can add ~34KB gzip. Tree-shake aggressively. Import only what is used.

// Example: branch card stagger reveal
animate(&apos;.branch-card&apos;, 
  { opacity: [0, 1], y: [8, 0] },
  { delay: stagger(0.06), duration: 0.22, easing: spring({ stiffness: 300, damping: 25 }) }
)

Fix 3 — Perceived Zero Latency (Dexie cache + prefetch)
The user should never see a loading state when they open the branch tree. The branches must already be there. This requires a background prefetch triggered by typing, not by the keyboard shortcut.

Prefetch trigger
800ms after typing stops (debounced). Background API call. User has not asked yet.
Cache location
Dexie.js IndexedDB table: branch_cache keyed by (session_id, draft_hash). TTL 5 minutes.
On shortcut open
Check cache first. If hit: show cached branches instantly with a subtle &apos;updated X seconds ago&apos; badge. If miss: show skeleton shimmer, resolve in <800ms.
Stale cache handling
Show stale branches immediately (dimmed, marked &apos;updating&apos;). Swap in fresh results when ready. No blank state ever.
Parameterization cache
Scoring config cached in Dexie by (session_id). Never regenerated mid-session unless preset changes.

3.3 The Annotation Glyph — Technical Spec
The chess annotation symbol is the most sensitive UI surface. It must appear as if it belongs to the compose surface, not painted on top of it.

Positioning
Relative to compose box bounding rect. Absolute positioned to top-right of active compose field. Not caret-adjacent at MVP (too fragile across editors). Caret-adjacent as v2 once stable.
Container
Small floating element injected into ShadowRoot. Not injected into the host compose DOM.
Symbol rendering
Large, weighted, using Geist Mono. Scale spring animation on appearance. Color coded: !! = blue, ! = green, !? = amber, ? = amber, ?? = red, - = neutral gray.
Appearance timing
Debounced 100ms after typing stops. Disappears when compose field loses focus.
Reason tooltip
Appears on hover over the symbol. One sentence. Floating UI positioned above symbol.
Sizing
Symbol: 18px Geist Mono, bold. Container: 28px × 28px. Subtle background circle.

The annotation scoring layer. Rules are generated by the parameterization call and evaluated locally on every draft change. Zero API calls on the hot path.

4.3 Memory Architecture — Phase 4
pgvector
PostgreSQL license — vector similarity search inside existing Cloud SQL. No new datastore. Add embedding column to interactions and new persona_shards table.
mem0ai/mem0
Apache-2.0 — memory policy layer. search(query, user_id) retrieves relevant persona shards. add(messages, user_id) creates memories from interactions. mem0-ts directory confirms TypeScript support.
Few-shot library
pgvector table of curated conversation examples. Embedded with text-embedding-3-small. Retrieved at parameterization time by cosine similarity.
What this replaces
Full persona JSONB blob injection on every call. Selective shard retrieval reduces tokens 30-50% while improving relevance.
Entry condition

4.4 Context Understanding — Phase 3
transformers.js
Apache-2.0 — local ONNX inference in Node API. Run sentiment + emotion pre-classification on recipient&apos;s last message before LLM context extraction call. Reduces entropy before paying tokens.
microsoft/RecognizersText
MIT — deterministic entity extraction. Dates, times, amounts, deadlines as structured fields. Injected into context extraction prompt as trusted data.
aurelio-labs/semantic-router
MIT — embedding-based dialogue act classification. Classify short messages (&apos;k&apos;, &apos;sure&apos;, &apos;let&apos;s talk&apos;) into actionable dialogue states without LLM call.

Package
philihp/openskill.js (MIT)
Source

Bayesian Weng-Lin rating. Models uncertainty (mu, sigma). Handles sparse interactions better than plain ELO. Supports recipient difficulty weighting.
Difficulty mapping
stranger > acquaintance > colleague for relationship. cold_outreach > linkedin > dating_app for platform. negotiate > pitch > date for stakes.

4.6 Branch Intelligence — Phase 6
LangGraphJS
MIT — graph-based workflow orchestration for the full extract → model → branch → simulate → rank pipeline
Tree-of-Thought pattern

Lookahead simulation
For each branch: generate message → run recipient model call → simulate one response → score 2-move sequence. Predicted responses become actual inference, not assertions.
Entry condition

Section 7  ·  Anti-Drift Governance
Rules that keep the system clean as it grows

The biggest risk is not technical. It is drift — features added without respecting the stage pipeline, infrastructure added without ADRs, prompt contracts changed without updating schemas, distribution logic in the application kernel. This section defines rules that prevent it.

7.1 The Six Drift Failure Modes
Failure mode
How it happens
Prevention
Stage pipeline violation
New feature bypasses context_extractor or persona_loader to save a round trip.
Every inference path goes through all stages. Stage skip requires ADR.
Scoring disconnected from pipeline
Annotation layer evaluates message quality instead of strategic fit against goal + recipient + persona.
Parameterization config is the session&apos;s source of truth. Rule engine consumes it. LLM arbitration only on ambiguity.
Distribution in product
Social posting, persona account config, or AI-OS workflow code appears in apps/persona1-ext or apps/persona1-api.
Hard boundary: distribution lives exclusively in businesses/persona1/social-*.yaml. Zero imports from distribution in application kernel.
Schema drift
Branch output schema changes in BAML but UI render code not updated. Or persona JSON changes but persona_updater prompt still references old fields.
Schema changes require: BAML update + migration + UI update + prompt update. All four in same PR.
Doc/code divergence
current-state.md says something is implemented but the code still runs a stub.
Every PR changing behavior includes a docs/ update. current-state.md reviewed on every merge.
Dependency sprawl
New package added without ADR. Six months later nobody knows why.
Every new package-level dependency requires ADR before PR merges.

7.2 PR Checklist — Mandatory Before Every Merge
Does this PR stay within its declared phase? No Phase 4 infra in a Phase 1 PR.
If adding a new dependency: is the ADR written and merged first?
If changing a prompt contract: are BAML schema, UI render code, and prompt all updated in this PR?
If changing persona JSON structure: is the migration written and persona_updater prompt updated?
If adding scoring rule: is it in the rule engine, not in the annotation prompt?
If touching distribution logic: does it stay in businesses/persona1? Is the application kernel unaware of it?
Is current-state.md still accurate after this merge?
Does corepack pnpm test still pass?
Is extension validation still green?
Is the Shadow DOM isolation intact? No extension styles escaping to host page?

7.3 The Canonical Pipeline Contract
Stage
Contract
context_extractor
Input: raw page DOM. Output: ContextOutput (BAML-typed). Runs once per session, cached. Fallback: manual paste.
parameterizer
Input: PersonaProfile + ContextOutput + preset. Output: ScoringConfig (BAML-typed). Runs once per session, cached in Dexie. Drives annotation layer for entire session.
persona_loader
Input: user_id + ContextOutput. Output: calibrated PersonaProfile. Phase 4+: retrieves relevant shards via mem0, not full blob. Fallback: full profile.
branch_generator
Input: PersonaProfile + ContextOutput + draft + preset. Output: BranchTree (BAML-typed, exactly 3 branches). Never called without both persona and context.
annotation_engine
Input: ScoringConfig + current draft. Output: AnnotationOutput (symbol + reason + confidence). Runs deterministically via json-rules-engine. Zero API calls. LLM arbitration only at confidence < 0.6.
persona_updater (async)
Input: PersonaProfile + interaction + outcome signal. Output: updated PersonaProfile. Runs after outcome capture. Never blocks UI.
mirror_trigger (async)
Input: interaction log. Output: string[] of insights (may be empty). Fires at interaction_count % 25 === 0. Only surfaces patterns with 5+ data points.
