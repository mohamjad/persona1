Section 2  ·  The Scoring Problem
Why the annotation is disconnected and the full fix

2.1 Architecture
Three layers. Different cost. Different timing. Together they make the annotation feel accurate instead of arbitrary.

Layer 1 — Parameterization (once per conversation, ~1 API call)
One LLM call fires when the compose context loads. Outputs a ScoringConfig JSON that drives annotation for the entire session. Cached in Dexie. Never repeated mid-session unless preset changes.

{
  "primary_goal": "get a yes to the meeting",
  "conversation_phase": "close",
  "tone_target": "warm-direct",
  "blunder_patterns": ["apology opener", "defensive language", "too many options"],
  "mistake_patterns": ["hedging before direct ask", "passive voice on ask"],
  "interesting_patterns": ["reframe their concern", "callback to earlier thread"],
  "brilliant_patterns": ["direct ask with single reason", "setup that makes them ask"],
  "recipient_risk_factors": ["responds negatively to pressure"],
  "persona_risk_factors": ["user hedges before asks"]
}

Layer 2 — Real-Time Deterministic (every keystroke, zero API calls)
json-rules-engine takes the ScoringConfig and evaluates the current draft locally. Runs in under 5ms. Debounced 100ms. Returns symbol + reason + confidence.

Rule engine source
json-rules-engine  ·  MIT
GitHub: https://github.com/CacheControl/json-rules-engine
Source: Scoring architecture — not in PDFs, selected for TS compatibility
JSON-defined rule engine. Takes fact set, evaluates against rule config. Perfect for dynamic scoring configs generated by LLM parameterization.

Layer 3 — LLM Arbitration (only when confidence < 0.6)
When rule engine returns ambiguous signals, a narrow LLM call fires. Not 'is this good' but 'given these specific signals, confirm or override the annotation.' Keeps LLM on edge cases only.

2.2 Parameterization Prompt
SYSTEM: You are the scoring configuration engine for persona1.
Output a ScoringConfig JSON. No commentary.
Name exact linguistic and structural patterns the rule engine can check.

2.3 Few-Shot Training Library
Curated conversation examples stored in pgvector, retrieved at parameterization time by cosine similarity. Makes scoring grounded in real outcomes not abstract principles.

Negotiation: Never Split the Difference case studies, recorded sales calls, FBI hostage negotiation examples
Dating: documented conversations with outcome annotation, academic attraction/rapport literature
Professional: email case studies, pitch recordings with known outcomes
Conflict: mediation transcripts, Crucial Conversations and Radical Candor frameworks

Embedding model
text-embedding-3-small via OpenRouter — cheap, accurate for semantic similarity
Storage
pgvector table few_shot_examples in existing Cloud SQL
Retrieval
Top-k by cosine similarity to current preset + recipient archetype at parameterization time

50-100 curated examples per major preset. Grows from user outcomes over time.

3.1 The Three Highest-Leverage UX Fixes
Directly from UX research PDF synthesis section, p.10-11.

Shadow DOM isolation — moves all extension UI into a ShadowRoot. Single biggest cause of clunky is CSS collision. This fixes it permanently.
Deliberate motion language — AutoAnimate has a ceiling. Motion gives you spring physics, stagger timelines, and weighted glyph appearance. One consistent motion signature makes the product feel designed.
Perceived zero latency — Dexie branch prefetch plus progressive disclosure. Show cached branches instantly. Skeleton on cache miss. Never show blank state.

Section 7  ·  Anti-Drift Governance
Rules that keep the system clean as it grows

7.1 Six Drift Failure Modes
Failure mode
How it happens
Prevention
Stage pipeline violation
New feature bypasses context_extractor or persona_loader to save a round trip
Every inference path goes through all stages. Stage skip requires ADR.
Scoring disconnected
Annotation layer evaluates message quality instead of goal + recipient + persona
Parameterization config is session source of truth. Rule engine consumes it. LLM arbitration only at confidence < 0.6.
Distribution in product
Social posting or AI-OS logic appears in apps/persona1-ext or apps/persona1-api
Hard boundary: distribution lives exclusively in businesses/persona1. Zero imports in application kernel.
Schema drift
Branch output schema changes in BAML but UI render code not updated
Schema changes require BAML update + migration + UI update + prompt update. All four in same PR.
Doc/code divergence
current-state.md says something is implemented but code runs a stub
Every PR changing behavior includes docs/ update. current-state.md reviewed on every merge.
Dependency sprawl
New package added without ADR
Every new package-level dependency requires ADR before PR merges.

7.2 Mandatory PR Checklist
Does this PR stay within its declared phase?
If adding a new dependency: is the ADR written and merged first?
If changing a prompt contract: are BAML schema, UI render code, and prompt all updated?
If changing persona JSON structure: is the migration written and persona_updater prompt updated?
If adding a scoring rule: is it in json-rules-engine, not in the annotation prompt?
If touching distribution logic: does it stay in businesses/persona1?
Is current-state.md still accurate?
Does corepack pnpm test still pass?
Is extension validation still green?
Is Shadow DOM isolation intact?

7.3 Canonical Pipeline Contract
Stage
Contract
context_extractor
Input: raw DOM. Output: ContextOutput (BAML-typed). Cached per session. Fallback: manual paste.
parameterizer
Input: PersonaProfile + ContextOutput + preset. Output: ScoringConfig (BAML-typed). Cached in Dexie by session_id. Drives annotation for entire session.
persona_loader
Input: user_id + ContextOutput. Output: calibrated PersonaProfile. Phase 4+: mem0 selective shard retrieval. Fallback: full profile blob.
branch_generator
Input: PersonaProfile + ContextOutput + draft + preset. Output: BranchTree (BAML-typed, exactly 3 branches). Prefetched 800ms after typing stops.
annotation_engine
Input: ScoringConfig + current draft. Output: AnnotationOutput (symbol + reason + confidence). Deterministic via json-rules-engine. Zero API calls. LLM arbitration only at confidence < 0.6.
persona_updater (async)
Input: PersonaProfile + interaction + outcome. Output: updated PersonaProfile. Never blocks UI. Fallback: deterministic update.
mirror_trigger (async)
Input: interaction log. Output: string[] insights. Fires at interaction_count % 25 === 0. Only surfaces patterns with 5+ data points.
